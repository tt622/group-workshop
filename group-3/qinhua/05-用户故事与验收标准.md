> **本文只做一件事**：按用户能理解的子场景拆成多条 **User Story**，并为每条写清 **REQ-ID**、**验收标准（AC）**。

## 字段说明 — 每条 UserStory 必须包含以下字段

| 字段 | 填写说明 | 数据来源 ← | 流向下游 → |
|------|----------|------------|-----------|
| **需求编号** | 唯一编号，如 REQ-M1QA-001 | ← `02` 采集编号 | → `14` 追踪矩阵首列 |
| **故事** | 作为【角色】，我希望【行为】，以便【价值】 | ← `04` PRD 场景拆解 | → `06` FSD 功能点 |
| **优先级** | P0（必须）/ P1（应该）/ P2（可选） | ← `03` §4 成功标准 | → `12` 里程碑排期 |
| **来源** | 对应 03/04 哪一条需求 | ← `03` 范围 · `04` 场景 | |
| **前置条件** | 触发该故事的前提状态 | ← `04` 用户流程 | → `13` 测试前置 |
| **功能对齐** | 对应功能规格 + 非功能约束 | | → `06` FSD · `07` NFR |
| **设计对齐** | 对应架构、接口、数据、安全 | | → `08` 架构 · `09` API · `10` Data · `11` 安全 |
| **验证对齐** | 对应测试用例 + 追踪矩阵行 | | → `13` TC · `14` 追踪矩阵行 |
| **验收标准** | AC，每条可独立验证 | ← 故事 + `07` NFR | → `13` 测试用例依据 |
