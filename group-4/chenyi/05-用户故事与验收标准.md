> **本文只做一件事**：按用户能理解的子场景拆成多条 **User Story**，并为每条写清 **REQ-ID**、**验收标准（AC）**。

## 字段说明 — 每条 UserStory 必须包含以下字段

| 字段 | 填写说明 | 数据来源 ← | 流向下游 → |
|------|----------|------------|-----------|
| **需求编号** | 唯一编号，如 REQ-M1QA-001 | ← `02` 采集编号 | → `14` 追踪矩阵首列 |
| **故事** | 作为【角色】，我希望【行为】，以便【价值】 | ← `04` PRD 场景拆解 | → `06` FSD 功能点 |
| **优先级** | P0（必须）/ P1（应该）/ P2（可选） | ← `03` §4 成功标准 | → `12` 里程碑排期 |
| **来源** | 对应 03/04 哪一条需求 | ← `03` 范围 · `04` 场景 | |
| **前置条件** | 触发该故事的前提状态 | ← `04` 用户流程 | → `13` 测试前置 |
| **功能对齐** | 对应功能规格 + 非功能约束 | | → `06` FSD · `07` NFR |
| **设计对齐** | 对应架构、接口、数据、安全 | | → `08` 架构 · `09` API · `10` Data · `11` 安全 |
| **验证对齐** | 对应测试用例 + 追踪矩阵行 | | → `13` TC · `14` 追踪矩阵行 |
| **验收标准** | AC，每条可独立验证 | ← 故事 + `07` NFR | → `13` 测试用例依据 |

## US-001（P0）会话管理

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M1QA-001 |
| **故事** | 作为投研经理，我希望能新建、列表查看、删除会话，以便管理不同研报或主题的问答上下文，避免信息混淆 |
| **优先级** | P0 |
| **来源** | ← 03 §3 范围内"会话管理" / 04 §3 SC-01、SC-05 / 04 §5 能力优先级 |
| **前置条件** | 用户已进入系统主界面；后端服务正常启动 |
| **功能对齐** | → 06 §会话管理功能规格 · 07 §响应时间约束 |
| **设计对齐** | → 08 §会话模块架构 · 09 POST /sessions · GET /sessions · DELETE /sessions/{id} · 10 sessions 表 · 11 §数据隔离 |
| **验证对齐** | → 13 TC-001 · 14 第 1 行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-001-01 | 点击"新建会话"后，系统创建会话并返回唯一 `session_id`，会话列表中出现该新条目，响应时间 ≤ 1 秒 |
| AC-001-02 | 会话列表按创建时间倒序展示，仅显示状态为"活跃"的会话，已删除会话不出现在列表中 |
| AC-001-03 | 删除会话后，该会话从列表消失；再次访问该 `session_id` 的接口返回 404，原始数据未物理清除（软删除） |

## US-002（P0）问答提交

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M1QA-002 |
| **故事** | 作为投研经理，我希望在会话中输入自然语言问题并获得带原文引用的回答，以便快速从研报中提取关键判断，无需手动翻阅全文 |
| **优先级** | P0 |
| **来源** | ← 03 §3 范围内"自然语言问答" / 04 §3 SC-01、SC-03 / 04 §4 R-01～R-04 |
| **前置条件** | 已创建会话；至少上传并完成解析一份研报 |
| **功能对齐** | → 06 §问答提交功能规格 · 07 §响应时间、降级约束 |
| **设计对齐** | → 08 §LLM 调用与降级模块 · 09 POST /sessions/{id}/messages · 10 messages 表 · 11 §本地部署数据不出域 |
| **验证对齐** | → 13 TC-002 · 14 第 2 行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-002-01 | 提交长度 1～500 字符的 `query`，系统返回包含 `answer`、`answer_source`、`cited_passages`、`session_id`、`created_at` 的完整响应 |
| AC-002-02 | 提交空字符串或超过 500 字符的 `query`，系统返回明确错误信息（HTTP 400），不触发 LLM 调用 |
| AC-002-03 | 外部 LLM 不可用时，系统依次降级至本地模型（`answer_source=local`），再不可用时降级至 Demo 模式（`answer_source=demo`），界面显示当前模式提示，不出现空白页或未处理异常 |

## US-003（P0）历史记录

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M1QA-003 |
| **故事** | 作为投研经理，我希望随时查看某次会话的完整问答历史，以便复盘研判过程、追溯信息来源 |
| **优先级** | P0 |
| **来源** | ← 03 §4 成功标准"主路径可走通" / 04 §3 SC-05 / 04 §4 R-05 |
| **前置条件** | 已有至少一条问答记录存在于该会话中 |
| **功能对齐** | → 06 §历史记录展示功能规格 · 07 §数据持久化约束 |
| **设计对齐** | → 08 §数据存储模块 · 09 GET /sessions/{id}/messages · 10 messages 表 · 11 §历史数据只读保护 |
| **验证对齐** | → 13 TC-003 · 14 第 3 行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-003-01 | 进入已有问答的会话，历史消息按时间正序完整展示，每条记录包含提问内容、回答内容、`answer_source` 标识及时间戳 |
| AC-003-02 | 刷新页面或重新打开系统后，历史记录仍完整保留，无数据丢失（持久化验证） |

## US-004（P1）健康检查

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M1QA-004 |
| **故事** | 作为 IT 运维人员，我希望通过统一接口查询系统各组件（后端服务、本地模型、存储层）的运行状态，以便在服务异常时快速定位问题并触发降级预案 |
| **优先级** | P1 |
| **来源** | ← 03 §5 风险"外部 LLM 不可用 / 本地模型推理慢" / 04 §4 R-04 降级策略 |
| **前置条件** | 系统已完成本地部署，后端服务已启动 |
| **功能对齐** | → 06 §健康检查功能规格 · 07 §可用性、响应时间约束 |
| **设计对齐** | → 08 §服务监控模块 · 09 GET /health · 10 无持久化要求 · 11 §接口访问控制（内网限制） |
| **验证对齐** | → 13 TC-004 · 14 第 4 行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-004-01 | 调用 `GET /health`，正常状态下 1 秒内返回各组件状态（后端服务、本地模型、存储层），HTTP 200，结构包含 `status: ok` 及各子项状态字段 |
| AC-004-02 | 模拟本地模型不可用时，`GET /health` 仍返回 HTTP 200，但对应子项状态标记为 `degraded`，不因单一组件异常导致整体接口报错 |
