## US-001（P0）会话创建与上下文管理

| 字段 | 内容 |
|------|------|
| **需求编号** | `REQ-M1QA-001` |
| **故事** | 作为投研分析师，我希望创建、切换和删除问答会话，以便按研究主题组织研报、问题和对比结果。 |
| **优先级** | P0 |
| **来源** | `03` §3 范围边界“会话管理：创建、列表、删除” · `03` §4 主路径可走通 |
| **前置条件** | 用户已进入系统首页，前后端服务可用。 |
| **功能对齐** | → `06` 会话侧栏 / 当前上下文切换（待同步） · `07` 基本输入与状态约束 |
| **设计对齐** | → `08` 前后端分层 · `09` sessions 相关接口（待同步） · `10` Session 数据模型（待同步） · `11` 会话数据访问控制 |
| **验证对齐** | → `13` 会话 CRUD 测试（待同步） · `14` 会话管理需求行（待同步） |

| 验收标准 | 描述 |
|----------|------|
| AC-001-01 | 用户进入系统后，系统 MUST 展示会话列表，并允许创建新会话。 |
| AC-001-02 | 用户点击“新建会话”后，系统 MUST 创建一个空会话并自动切换到该会话上下文。 |
| AC-001-03 | 用户切换会话时，系统 MUST 仅展示该会话关联的研报、问答与对比记录，不得串线。 |
| AC-001-04 | 用户删除会话时，系统 MUST 提示确认；确认后删除该会话及其关联上下文数据。 |

## US-002（P0）导入并解析多格式研报

| 字段 | 内容 |
|------|------|
| **需求编号** | `REQ-M1QA-002` |
| **故事** | 作为投研分析师，我希望导入 WORD、PPT、PDF、HTML 格式的研报，并由系统自动解析为可读取内容，以便后续提取、问答和对比。 |
| **优先级** | P0 |
| **来源** | `02` S-01 需求记录“研报解析” · `03` §2 提案概要 · `03` §3 范围边界“支持 WORD/PPT/PDF/HTML” |
| **前置条件** | 用户已进入某个会话，且拥有待解析研报文件。 |
| **功能对齐** | → `06` 研报上传区 / 文档列表 / 解析状态（待同步） · `07` 文件格式、大小、解析耗时约束 |
| **设计对齐** | → `08` 文档解析链路（待同步） · `09` 研报上传与解析接口（待同步） · `10` Report 文档实体（待同步） · `11` 文件安全与上传校验 |
| **验证对齐** | → `13` 多格式解析与失败回退测试（待同步） · `14` 研报解析需求行（待同步） |

| 验收标准 | 描述 |
|----------|------|
| AC-002-01 | 系统 MUST 支持上传 `WORD / PPT / PDF / HTML` 四类研报文件，并在上传后进入解析流程。 |
| AC-002-02 | 解析成功后，系统 MUST 生成可供展示和后续计算的文本内容；解析失败时 MUST 返回明确失败状态和原因。 |
| AC-002-03 | 单个文件解析失败 MUST NOT 导致当前会话中其他已导入研报不可用。 |
| AC-002-04 | 对于文本层不足的 PDF，系统 SHOULD 提供 OCR 兜底路径，以降低解析失败率。 |

## US-003（P0）查看结构化提取结果

| 字段 | 内容 |
|------|------|
| **需求编号** | `REQ-M1QA-003` |
| **故事** | 作为投研分析师，我希望系统自动提取研报中的评级、目标价和核心观点，并以结构化方式展示，以便快速抓住研报重点。 |
| **优先级** | P0 |
| **来源** | `02` S-01/S-02 “提取标题、评级、目标价、核心观点” · `03` §2 提案概要 · `03` §4 成功标准“提取准确率 ≥ 90%” |
| **前置条件** | 至少一份研报已完成解析。 |
| **功能对齐** | → `06` 结构化摘要卡片 / 来源高亮（待同步） · `07` 提取准确率与缺失字段策略 |
| **设计对齐** | → `08` 提取流水线（待同步） · `09` 研报摘要查询接口（待同步） · `10` 结构化字段模型（待同步） · `11` 结果审计与留痕 |
| **验证对齐** | → `13` 提取正确率与空值处理测试（待同步） · `14` 结构化提取需求行（待同步） |

| 验收标准 | 描述 |
|----------|------|
| AC-003-01 | 对已解析研报，系统 MUST 输出至少以下结构化字段：`评级`、`目标价`、`核心观点`；无法识别时 MUST 明确标记“未提取到”，不得臆造。 |
| AC-003-02 | 每个结构化字段 SHOULD 能追溯到原文片段或来源位置，便于人工复核。 |
| AC-003-03 | 在 20 份测试研报的验收样本下，评级、目标价、核心观点的综合提取准确率 SHOULD 达到 `90%` 及以上。 |

## US-004（P0）基于研报内容提问并引用原文作答

| 字段 | 内容 |
|------|------|
| **需求编号** | `REQ-M1QA-004` |
| **故事** | 作为投研分析师，我希望围绕已导入研报提出自然语言问题，并获得带原文引用的回答，以便快速确认投资观点与分析依据。 |
| **优先级** | P0 |
| **来源** | `02` S-01 “基于研报内容进行自然语言问答，支持引用原文” · `03` §2 提案概要 · `03` §4 主路径可走通 |
| **前置条件** | 当前会话已导入至少一份可用研报。 |
| **功能对齐** | → `06` 问答区 / 引用展示 / 历史记录（待同步） · `07` 输入约束与响应时延 |
| **设计对齐** | → `08` 问答编排链路（待同步） · `09` 问答接口（待同步） · `10` QARecord 模型（待同步） · `11` 引用内容脱敏与输出安全 |
| **验证对齐** | → `13` 问答 happy path / 引用有效性 / 输入校验测试（待同步） · `14` 问答需求行（待同步） |

| 验收标准 | 描述 |
|----------|------|
| AC-004-01 | 用户 MUST 只能在会话中对已导入研报发起问题；问题提交后，系统 MUST 返回回答并写入当前会话历史。 |
| AC-004-02 | 每次回答 MUST 附带至少一处可见的原文引用、出处片段或来源标识。 |
| AC-004-03 | 当当前研报中缺少支撑证据时，系统 MUST 明确说明“未在当前研报中找到依据”或等价提示，不得伪造引用。 |
| AC-004-04 | 用户在 3 分钟内 MUST 能完成“新建会话 → 导入研报 → 提问 → 查看历史”的主路径闭环。 |

## US-005（P0）对比多份研报关键指标

| 字段 | 内容 |
|------|------|
| **需求编号** | `REQ-M1QA-005` |
| **故事** | 作为投研分析师，我希望选择多份关于同一主题或标的的研报并自动生成关键指标对比结果，以便快速发现观点差异。 |
| **优先级** | P0 |
| **来源** | `02` S-01/S-02 “多份研报对比困难，希望自动生成对比表” · `03` §2 提案概要 · `03` §4 成功标准“2 分钟内生成对比列表” |
| **前置条件** | 当前会话中已存在两份及以上可解析研报。 |
| **功能对齐** | → `06` 研报选择器 / 对比结果表格（待同步） · `07` 批量处理时延约束 |
| **设计对齐** | → `08` 对比编排与字段映射（待同步） · `09` 对比接口（待同步） · `10` CompareResult 模型（待同步） · `11` 多文档访问授权 |
| **验证对齐** | → `13` 双文档/多文档对比测试（待同步） · `14` 对比能力需求行（待同步） |

| 验收标准 | 描述 |
|----------|------|
| AC-005-01 | 用户选择两份或以上研报后，系统 MUST 生成结构化对比结果，至少覆盖 `评级`、`目标价`、`核心观点`。 |
| AC-005-02 | 对比结果 MUST 保留每个字段的来源研报标识，便于用户识别差异来源。 |
| AC-005-03 | 当某份研报缺失某个对比字段时，系统 MUST 明确展示为空或“未提取到”，不得用其他研报内容代填。 |
| AC-005-04 | 上传两份研报后，系统 SHOULD 在 `2 分钟` 内生成可查看的对比列表。 |

## US-006（P0）查询研报关联股票信息

| 字段 | 内容 |
|------|------|
| **需求编号** | `REQ-M1QA-006` |
| **故事** | 作为投研分析师，我希望在阅读研报时直接查询关联股票信息，以便不离开当前工作台就完成“读报 + 查数据”的连续动作。 |
| **优先级** | P0 |
| **来源** | `02` S-02 “在研报阅读界面直接查到相关股票数据” · `03` §2 提案概要 · `03` §4 成功标准“推荐查询在研报中出现的股票代码，可查询全市场股票信息” |
| **前置条件** | 用户已打开某份研报或已识别出相关标的。 |
| **功能对齐** | → `06` 股票检索入口 / 结果面板 / 研报联动（待同步） · `07` 查询性能与异常提示 |
| **设计对齐** | → `08` 股票数据接入层（待同步） · `09` 股票查询接口（待同步） · `10` StockSnapshot DTO（待同步） · `11` 外部数据源接入安全 |
| **验证对齐** | → `13` 股票查询与联动展示测试（待同步） · `14` 股票查询需求行（待同步） |

| 验收标准 | 描述 |
|----------|------|
| AC-006-01 | 系统 MUST 提供从研报上下文发起股票查询的入口，支持基于代码、名称或研报中识别出的标的进行查询。 |
| AC-006-02 | 对于可识别的标的，系统 MUST 返回可阅读的股票基础信息或行情信息，并在当前界面展示。 |
| AC-006-03 | 当查询命中多个候选标的时，系统 MUST 允许用户明确选择目标股票后再查看详情。 |
| AC-006-04 | 股票查询 MUST 不要求用户切换到外部系统才能完成。 |

## US-007（P1）外部模型异常时保持友好可用

| 字段 | 内容 |
|------|------|
| **需求编号** | `REQ-M1QA-007` |
| **故事** | 作为投研分析师，我希望在外部 LLM 不可用时系统仍能有清晰、可接受的处理方式，以便关键流程不会因为上游故障而完全中断。 |
| **优先级** | P1 |
| **来源** | `03` §2 提案概要“三级降级保障” · `03` §5 风险“外部 LLM 不可用” · `03` §3 范围外“降级保障（先默认外部llm可用）” |
| **前置条件** | 问答或提取流程依赖外部模型，且上游模型不可用或超时。 |
| **功能对齐** | → `06` 异常提示 / 能力状态反馈（待同步） · `07` 可用性与降级策略 |
| **设计对齐** | → `08` 模型编排与 fallback（待同步） · `09` 错误码与状态接口（待同步） · `11` 上游错误脱敏 |
| **验证对齐** | → `13` 上游异常与降级分支测试（待同步） · `14` 可用性需求行（待同步） |

| 验收标准 | 描述 |
|----------|------|
| AC-007-01 | 若当前部署启用了降级链路，系统 SHOULD 按预设顺序执行降级，并返回明确的来源标识。 |
| AC-007-02 | 若当前版本未实现完整降级链路，系统 MUST 向用户返回可理解的失败提示，并保留当前会话与已上传研报数据。 |
| AC-007-03 | 无论走降级还是失败提示路径，系统 MUST NOT 暴露密钥、内部 URL、调用栈等敏感细节。 |
