> **本文只做一件事**：按用户能理解的子场景拆成多条 **User Story**，并为每条写清 **REQ-ID**、**验收标准（AC）**。

## 字段说明 — 每条 UserStory 必须包含以下字段

| 字段 | 填写说明 | 数据来源 ← | 流向下游 → |
|------|----------|------------|-----------|
| **需求编号** | 唯一编号，如 REQ-M1QA-001 | ← `02` 采集编号 | → `14` 追踪矩阵首列 |
| **故事** | 作为【角色】，我希望【行为】，以便【价值】 | ← `04` PRD 场景拆解 | → `06` FSD 功能点 |
| **优先级** | P0（必须）/ P1（应该）/ P2（可选） | ← `03` §4 成功标准 | → `12` 里程碑排期 |
| **来源** | 对应 03/04 哪一条需求 | ← `03` 范围 · `04` 场景 | |
| **前置条件** | 触发该故事的前提状态 | ← `04` 用户流程 | → `13` 测试前置 |
| **功能对齐** | 对应功能规格 + 非功能约束 | | → `06` FSD · `07` NFR |
| **设计对齐** | 对应架构、接口、数据、安全 | | → `08` 架构 · `09` API · `10` Data · `11` 安全 |
| **验证对齐** | 对应测试用例 + 追踪矩阵行 | | → `13` TC · `14` 追踪矩阵行 |
| **验收标准** | AC，每条可独立验证 | ← 故事 + `07` NFR | → `13` 测试用例依据 |

## US-001（P0）会话管理

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M1QA-001 |
| **故事** | 作为高级分析师，我希望可以对比研报的指标和内容区分 |
| **优先级** | P0 |
| **来源** | ← 03 §2.1 |
| **前置条件** | 用户已登录系统 |
| **功能对齐** | → 06 §支持在页面新建/删除会话，且可以为会话自动命名或者改会话名 · 07 §创建会话速度要在1秒以内 |
| **设计对齐** | → 08 §采用前后端分离架构，会话记录存储在浏览器端 · 09 §暂时不需要API接口，仅需前端实现会话创建/删除/命名/编辑名称 · 10 §不依赖数据 · 11 §暂无 |
| **验证对齐** | → 13 TC-测试会话创建/删除/命名/编辑名称 · 14 第__行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-001-01 | 用户可以在页面新建会话 |
| AC-001-02 | 用户可以删除会话 |
| AC-001-03 | 用户可以为会话自动命名或修改会话名称 |

## US-002（P0）问答提交

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M1QA-002 |
| **故事** | 作为高级分析师，我希望可以对研报做问答，作为第一阶段，先实现简单的问答 |
| **优先级** | P0 |
| **来源** | ← 03 §2.1 |
| **前置条件** | 已创建会话 |
| **功能对齐** | → 06 §支持输入文本内容，并提交问题获得流式回复 · 07 §首字符回复在5秒内 |
| **设计对齐** | → 08 §问题会提交到后端去处理，后端会用流式返回结果 · 09 §/v1/chat API（SSE） · 10 §第一阶段仅作简单问答，后续依赖RAG数据源需召回排序后提供 · 11 §问题需要接入远程日志留痕 |
| **验证对齐** | → 13 TC-测试短/中/长问题一次性提问及多轮问答 · 14 第__行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-002-01 | 用户可以输入文本内容并提交问题 |
| AC-002-02 | 系统返回流式回复，首字符回复在5秒内 |
| AC-002-03 | 支持多轮问答 |

## US-003（P0）知识库研报上传构建

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M1QA-003 |
| **故事** | 作为高级分析师，我希望可以上传研报用于后续的研报问答 |
| **优先级** | P0 |
| **来源** | ← 03 §2.1 |
| **前置条件** | 用户已登录系统 |
| **功能对齐** | → 06 §支持输入为研报文件，并可以解析文件、切块、向量化存储起来 · 07 §上传入库单文件2秒内 |
| **设计对齐** | → 08 §前端提供上传文件按钮，看到任务执行状态的能力；后端支持解析文件、切块、向量化存储至milvus · 09 §/v1/rag/upload API · 10 §输入的研报数据 · 11 §日志留痕 |
| **验证对齐** | → 13 TC-测试小/中/大研报文件上传，覆盖excel、word、pdf等多类格式 · 14 第__行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-003-01 | 用户可以上传研报文件 |
| AC-003-02 | 系统可以解析文件、切块、向量化存储 |
| AC-003-03 | 上传入库单文件2秒内完成 |
