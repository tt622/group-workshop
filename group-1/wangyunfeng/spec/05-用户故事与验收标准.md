> **本文只做一件事**：按用户能理解的子场景拆成多条 **User Story**，并为每条写清 **REQ-ID**、**验收标准（AC）**。

## 字段说明 — 每条 UserStory 必须包含以下字段

| 字段 | 填写说明 | 数据来源 ← | 流向下游 → |
|------|----------|------------|-----------|
| **需求编号** | 唯一编号，如 REQ-M1QA-001 | ← `02` 采集编号 | → `14` 追踪矩阵首列 |
| **故事** | 作为【角色】，我希望【行为】，以便【价值】 | ← `04` PRD 场景拆解 | → `06` FSD 功能点 |
| **优先级** | P0（必须）/ P1（应该）/ P2（可选） | ← `03` §4 成功标准 | → `12` 里程碑排期 |
| **来源** | 对应 03/04 哪一条需求 | ← `03` 范围 · `04` 场景 | |
| **前置条件** | 触发该故事的前提状态 | ← `04` 用户流程 | → `13` 测试前置 |
| **功能对齐** | 对应功能规格 + 非功能约束 | | → `06` FSD · `07` NFR |
| **设计对齐** | 对应架构、接口、数据、安全 | | → `08` 架构 · `09` API · `10` Data · `11` 安全 |
| **验证对齐** | 对应测试用例 + 追踪矩阵行 | | → `13` TC · `14` 追踪矩阵行 |
| **验收标准** | AC，每条可独立验证 | ← 故事 + `07` NFR | → `13` 测试用例依据 |

## US-001（P0）会话管理

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M1QA-001 |
| **故事** | 作为研究员，我希望创建、查看和删除会话，以便管理我的研报问答记录 |
| **优先级** | P0 |
| **来源** | ← 03 §3 范围边界（会话管理：创建、列表、删除） |
| **前置条件** | 用户已访问系统 |
| **功能对齐** | → 06 §会话管理 · 07 §可用性 |
| **设计对齐** | → 08 §架构设计 · 09 /sessions API · 10 sessions表 · 11 §访问控制 |
| **验证对齐** | → 13 TC-001 · 14 第1行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-001-01 | 用户可以创建新会话，系统返回唯一会话ID |
| AC-001-02 | 用户可以查看所有会话列表，按时间倒序排列 |
| AC-001-03 | 用户可以删除指定会话及其关联的问答记录 |

## US-002（P0）问答提交

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M1QA-002 |
| **故事** | 作为研究员，我希望提交研报相关问题并获取关键要素提取结果，以便快速了解研报核心内容 |
| **优先级** | P0 |
| **来源** | ← 03 §3 范围边界（要素提取，比如评级、目标价格、核心观点等） |
| **前置条件** | 已创建会话 |
| **功能对齐** | → 06 §问答提交 · 07 §性能要求 |
| **设计对齐** | → 08 §架构设计 · 09 /chat API · 10 messages表 · 11 §数据安全 |
| **验证对齐** | → 13 TC-002 · 14 第2行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-002-01 | 用户可以在1分钟内获取研报关键要素（评级、目标价格、核心观点） |
| AC-002-02 | 系统支持多轮问答，保持会话上下文 |
| AC-002-03 | 当外部LLM不可用时，系统按三级降级策略（CoPaw → 百炼 → Demo模式）返回结果 |
