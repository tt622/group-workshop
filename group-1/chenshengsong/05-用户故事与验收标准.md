> **本文只做一件事**：按用户能理解的子场景拆成多条 **User Story**，并为每条写清 **REQ-ID**、**验收标准（AC）**。

## 字段说明 — 每条 UserStory 必须包含以下字段

| 字段 | 填写说明 | 数据来源 ← | 流向下游 → |
|------|----------|------------|-----------|
| **需求编号** | 唯一编号，如 REQ-M1QA-001 | ← `02` 采集编号 | → `14` 追踪矩阵首列 |
| **故事** | 作为【角色】，我希望【行为】，以便【价值】 | ← `04` PRD 场景拆解 | → `06` FSD 功能点 |
| **优先级** | P0（必须）/ P1（应该）/ P2（可选） | ← `03` §4 成功标准 | → `12` 里程碑排期 |
| **来源** | 对应 03/04 哪一条需求 | ← `03` 范围 · `04` 场景 | |
| **前置条件** | 触发该故事的前提状态 | ← `04` 用户流程 | → `13` 测试前置 |
| **功能对齐** | 对应功能规格 + 非功能约束 | | → `06` FSD · `07` NFR |
| **设计对齐** | 对应架构、接口、数据、安全 | | → `08` 架构 · `09` API · `10` Data · `11` 安全 |
| **验证对齐** | 对应测试用例 + 追踪矩阵行 | | → `13` TC · `14` 追踪矩阵行 |
| **验收标准** | AC，每条可独立验证 | ← 故事 + `07` NFR | → `13` 测试用例依据 |

## US-001（P0）会话管理

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M1QA-001 |
| **故事** | 作为投研分析师，我希望创建和管理问答会话，以便按不同研究主题组织我的问题与答案 |
| **优先级** | P0 |
| **来源** | ← 03 §3 范围边界 / 04 §2 提案概要 |
| **前置条件** | 用户已登录系统 |
| **功能对齐** | → 06 §3.1 会话管理 · 07 §4.1 可用性 |
| **设计对齐** | → 08 §4 后端架构 · 09 /api/sessions API · 10 sessions表 · 11 §3 认证安全 |
| **验证对齐** | → 13 TC-001~003 · 14 第1-3行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-001-01 | 用户可创建新会话，系统自动生成会话ID和创建时间戳 |
| AC-001-02 | 用户可查看会话列表，按创建时间倒序排列，显示会话标题和最后更新时间 |
| AC-001-03 | 用户可删除会话，删除后该会话下所有问答记录一并删除，不可恢复 |

## US-002（P0）问答提交

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M1QA-002 |
| **故事** | 作为投研分析师，我希望上传研报并对研报内容进行自然语言问答，以便快速提取关键信息并获取深度解读 |
| **优先级** | P0 |
| **来源** | ← 03 §2 提案概要 / 04 §3 核心功能 |
| **前置条件** | 已创建会话，已上传至少一份研报（PDF/HTML格式） |
| **功能对齐** | → 06 §3.2 研报解析 · 06 §3.3 问答引擎 · 07 §3.1 性能要求 |
| **设计对齐** | → 08 §5 AI引擎 · 09 /api/qa/submit API · 10 qa_records表 · 11 §4 数据安全 |
| **验证对齐** | → 13 TC-004~007 · 14 第4-7行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-002-01 | 用户上传PDF/HTML研报后，系统在2分钟内自动提取标题、评级、目标价、核心观点 |
| AC-002-02 | 用户提交自然语言问题后，系统基于研报内容生成回答，并在80%以上的情况下正确引用原文 |
| AC-002-03 | 回答中必须标注引用来源（如："根据XX研报第3页..."），支持点击跳转到原文位置 |
| AC-002-04 | 当外部LLM不可用时，系统按三级降级策略（CoPaw → 百炼 → Demo模式）正常响应 |


## US-003（P1）研报对比

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M1QA-003 |
| **故事** | 作为投研分析师，我希望选择多份研报并生成关键指标对比表，以便快速评估不同券商的观点差异 |
| **优先级** | P1 |
| **来源** | ← 03 §2 提案概要 / 04 §3 核心功能 / 02 S-01 会议纪要 |
| **前置条件** | 已上传至少两份研报，且研报已解析完成 |
| **功能对齐** | → 06 §3.5 研报对比 · 07 §3.1 性能要求 |
| **设计对齐** | → 08 §5 AI引擎 · 09 /api/compare API · 10 compare_results表 · 11 §4 数据安全 |
| **验证对齐** | → 13 TC-010~011 · 14 第10-11行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-003-01 | 用户选择2-5份研报后，系统自动生成对比表，包含评级、目标价、核心观点、营收预测等关键指标 |
| AC-003-02 | 对比表生成时间在3分钟内完成，支持导出为Excel格式 |
| AC-003-03 | 对比表中不同券商的观点差异以高亮或颜色标注，便于快速识别 |

| **来源** | ← 02 S-01 会议纪要 / 02 S-02 邮件沟通 |

## US-005（P1）股票数据查询

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M1QA-005 |
| **故事** | 作为投研分析师，我希望在阅读研报时直接查询相关股票数据，以便快速获取行情信息而无需切换系统 |
| **优先级** | P1 |
| **来源** | ← 02 S-02 邮件沟通 / 02 S-03 用户访谈 |
| **前置条件** | 已打开研报或会话，研报中包含股票代码或名称 |
| **功能对齐** | → 06 §3.7 股票数据查询 · 07 §3.1 性能要求 |
| **设计对齐** | → 08 §6 数据采集模块 · 09 /api/stock API · 10 stock_data表 · 11 §4 数据安全 |
| **验证对齐** | → 13 TC-015~016 · 14 第15-16行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-005-01 | 系统自动识别研报中的股票代码/名称，提供一键查询入口 |
| AC-005-02 | 查询结果包含实时行情、历史走势、关键财务指标（PE、PB、营收等） |
| AC-005-03 | 股票数据查询响应时间在2秒内完成 |
